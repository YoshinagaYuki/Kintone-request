# allmight-request 要件定義書

- 作成日: 2026-07-03
- ステータス: ドラフト(レビュー待ち)
- 対象フェーズ: Phase 1(オールマイトのみ)

---

## 1. 概要

社外パートナー(Surely)が案件情報(FMT)を貼り付けて申請するための受付システム。
申請はいったん Supabase に「確認待ち」として保存し、ユニティ側の承認を経て kintone(オールマイトアプリ)へ登録、LINE WORKS へ通知する。

### 1.1 目的

- Surely からの案件申請を Web で受け付け、手作業の転記をなくす
- kintone へ直接登録させず、ユニティ側の確認・承認を挟むことでデータ品質を担保する
- 承認後の kintone 登録・LINE WORKS 通知を自動化する

### 1.2 スコープ

| 項目 | Phase 1 | 将来 |
|---|---|---|
| 対象案件種別 | オールマイトのみ | てずくーる等を追加 |
| 申請者 | Surely | 他パートナー追加の可能性 |
| kintone 連携先 | オールマイトアプリ (AppID: 10) | 案件種別ごとのアプリ |

## 2. 業務フロー

```
Surely
  │ 専用URLを開く(ログインなし)
  ▼
申請フォームで FMT を貼り付け
  │ 申請
  ▼
Supabase に「確認待ち」で保存
  │
  ▼
ユニティ側が管理画面で内容確認
  │
  ├─ 承認 ──► kintone(AppID:10)へ登録 ──► LINE WORKS へ通知
  │
  └─ 差戻し(理由を記録)
```

## 3. ユーザーと権限

| ユーザー | アクセス方法 | できること |
|---|---|---|
| Surely(申請者) | 専用URL(ログインなし・URLを知っていれば利用可) | FMT貼り付け → 申請のみ |
| ユニティ担当者(管理者) | 管理画面(Supabase Auth でログイン) | 申請一覧・内容確認・承認・差戻し |

- 申請側に認証は設けない。専用URLは推測困難なスラッグ/トークンとする
- 管理側は社内利用のためログイン必須とする

## 4. 機能要件

### 4.1 申請機能(Surely向け)

- FR-01: 専用URLで申請フォームを表示する(ログイン不要)
- FR-02: FMT(定型テキスト)をテキストエリアに貼り付けて申請できる
- FR-03: 申請時に FMT の形式チェック(必須項目の欠落など)を行い、エラーを表示する
- FR-04: 申請完了画面を表示する(受付番号等の識別子は表示しない。Supabase内部IDは管理用のみで、Surely側には見せない)
- FR-05: 申請データは Supabase に「確認待ち(pending)」で保存する。kintone へは直接登録しない

### 4.2 管理機能(ユニティ向け)

- FR-10: Supabase Auth によるログイン
- FR-11: 申請一覧をステータス・日付で絞り込み表示
- FR-12: 申請詳細で FMT 原文とパース結果を確認できる
- FR-13: 承認 → kintone(AppID:10)へレコード登録 → LINE WORKS へ通知
- FR-14: 差戻し → 理由を必須入力し記録
- FR-15: kintone 登録失敗時はステータスを「登録失敗」とし、再実行できる
- FR-16: 申請ごとの操作履歴(申請・承認・差戻し・登録・通知)を参照できる

### 4.3 連携機能

- FR-20: kintone REST API でオールマイトアプリ(AppID:10)へレコード登録
- FR-21: kintone 登録成功後、LINE WORKS Bot でトークルームへ通知
- FR-22: 登録した kintone レコードIDを Supabase 側に保持する

## 5. 非機能要件

- NFR-01: 申請URLは推測困難な文字列とし、robots/検索エンジンから除外する
- NFR-02: 申請APIにはレート制限を設け、スパム・二重送信を防止する
- NFR-03: kintone・LINE WORKS のAPIキーはサーバーサイドのみで保持する(クライアントへ露出しない)
- NFR-04: Supabase は RLS を有効化し、匿名からの読み取りを禁止する
- NFR-05: 操作履歴により監査可能とする

## 6. データ保持の役割分担

| システム | 役割 |
|---|---|
| Supabase | 受付システムのDB(一時保管ではない)。申請受付・確認待ちデータ・承認状態・差戻し理由・履歴管理 |
| kintone | 業務マスタおよび本番データ管理。正式案件・売上管理・発送管理・既存業務フロー |

## 7. 将来拡張の考慮

- 案件種別(フォーム種別)をマスタテーブルで管理し、「てずくーる」等を設定追加のみで増やせる構成にする
- 種別ごとに 専用URLスラッグ / FMTパース定義 / kintone AppID / フィールドマッピング / 通知先 を持たせる
- Phase 1 の実装はオールマイト固定でよいが、種別を前提にしたデータ構造にしておく

- 差戻し時に Surely へ連絡する方法(Phase 1 では手動連絡想定)
