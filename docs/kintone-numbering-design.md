# 管理番号採番・配送管理連携 設計書

- 作成日: 2026-07-03
- ステータス: 設計案(★要確認2点あり) / 実装は本設計に沿って完了
- 前提: 既存のkintone画面JS(保存イベントで採番+配送管理連携)はそのまま残す

---

## 1. 方針

承認時に allmight-request 側で、既存JSと**同じ採番マスタ・同じ手順**を REST API で実行する。
独自採番は行わない。

```
承認ボタン
  │
  ▼
[1] App10(オールマイト)へレコード登録            … 既存実装
  │
  ▼
[2] App50(採番マスタ)で採番                      … キー="shipping" の「現在番号」を +1
  │    revisionを使った楽観ロック(競合時リトライ最大5回)
  ▼
[3] App10 の管理番号フィールドを更新
  │
  ▼
[4] App11(配送管理)へ同じ管理番号で 新規作成 or 更新(upsert)
  │    既存JSと同じフィールド対応で転記
  ▼
status = registered
```

どこかで失敗した場合は status = register_failed + 履歴にエラー記録 → 画面の「再実行」で再開。

## 2. 冪等設計(再実行しても二重処理しない)

再実行(retry)は**途中から再開**できるよう、各ステップにスキップ条件を持つ:

| ステップ | スキップ条件 |
|---|---|
| [1] App10登録 | `requests.kintone_record_id` が既にある(登録直後に即保存する) |
| [2] 採番 | App10レコードの管理番号が既に入っている(既存画面JSと同じガード) |
| [3] App10更新 | [2]と一体(採番した場合のみ実行) |
| [4] App11 upsert | 常に実行(管理番号で検索し、あれば更新・なければ作成なので冪等) |

- 既存画面JSは「管理番号が既にある場合は採番しない」ため、本システムが先に採番すれば競合しない
- 逆に画面JSが先に採番済みでも[2]がスキップされるため二重採番しない

## 3. 採番アルゴリズム(App50)

1. `キー = "shipping"` のレコードを取得($id, $revision, 現在番号)
2. `新番号 = 現在番号 + 1`
3. revision 指定で 現在番号 を更新(PUT /k/v1/record.json)
4. revision不一致(HTTP 409)= 他プロセスが先に採番 → 1からやり直し(最大5回、待機を挟む)
5. 成功したら 新番号 を管理番号として使用

補足: [2]成功後に[3]以降が失敗した場合、その番号は欠番になる(再実行では新しい番号を採番)。
重複よりも欠番を許容する設計(採番マスタ方式の一般的なトレードオフ)。

## 4. 配送管理(App11)への転記

- `管理番号 = "<採番した番号>"` で App11 を検索
  - 0件 → 新規作成 / 1件 → 更新(既存JSの新規作成/更新と同じ挙動)
- 転記フィールドは `lib/kintone/shipping-mapping.ts` の対応表(App10コード → App11コード)で定義
- **★要確認A**: 既存JSのフィールド対応表が必要。既存JSファイルを共有いただければ同じ対応を設定する
  (未設定のまま承認すると明確なエラーメッセージで register_failed になり、設定後に再実行できる)

## 5. APIトークンと環境変数

kintoneのAPIトークンはアプリ単位のため追加が必要。リクエストには保有トークンを
カンマ結合で送る(kintone仕様: 最大9個)。

```bash
KINTONE_APP_ID_NUMBERING=50        # 採番マスタ
KINTONE_APP_ID_SHIPPING=11         # 配送管理
KINTONE_API_TOKEN=                 # App10(既存): フォーム参照+レコード追加+閲覧+編集
KINTONE_API_TOKEN_APP50=           # App50: レコード閲覧+編集
KINTONE_API_TOKEN_APP11=           # App11: レコード閲覧+追加+編集
```

※ App10トークンに「閲覧・編集」権限の追加が必要(管理番号の読み取り・書き込みのため)。

## 7. 履歴・ステータス

- 履歴アクションを追加: `numbered`(採番+App10更新完了)、`shipping_synced`(App11転記完了)
  → migration `0003_numbering.sql` で check制約を拡張

- 失敗はこれまで通り `kintone_failed`(detailにstepとエラー内容)+ status=register_failed

## 8. 既存JSとの関係

- 既存画面JSは変更せずそのまま残す(手入力レコードの採番・配送管理連携は従来通り)
- 本システム経由のレコードは登録時点で管理番号が入るため、画面JSは採番をスキップする
- 管理画面の登録完了表示は「採番・配送管理連携まで自動実行済み」の文面に変更
